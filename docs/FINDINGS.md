# FINDINGS (人手書き) — 確認された新事実・効いた施策

`ANTI_PATTERNS.md` の **正の対**。実験を進める中で「効いた」「真と確認できた」を、根拠つきで蓄積する。
EXP_SUMMARY (自動・機械的) では拾えない**横断的な洞察・新事実**を、人 (or AIエージェント) が言語化してここに残す。

> pre-flight では **ANTI_PATTERNS (避ける) と FINDINGS (活かす) を両方読む**。
> 失敗だけでなく「すでに効くと分かっていること」も制度的記憶にする。

## 0. 再評価メカニズム (FINDINGS ⇄ ANTI_PATTERNS)

判定には**有効期限**がある。新事実が片方に入ったら反対側の過去判定を再評価する:
- FINDINGS に新事実 → ANTI_PATTERNS の `🔓再評価トリガ` を満たす過去失敗を「再挑戦候補」に。
- ANTI_PATTERNS に新事実 → FINDINGS の `⚠️失効トリガ` を満たす成功を再検証。
- トリガは**具体的・反証可能**に (曖昧条件は thrash の元)。照合は LLM が追記時/pre-flight に行う (判断は自動化しない)。

## 効いた施策 (確認済み)

- **線形モデルには `standardize` が効く** — EXP001 で +0.00263 (5seed)。木では ±0.00000 (EXP000)。
  → 線形/NN メンバーには standardize を入れる。木には不要。
  ⚠️失効トリガ: 線形メンバーを使わなくなったら無効。🔓この事実は「tree で standardize 無駄=failed」を
  *線形に限り* 再評価させる (= EXP001 がまさにその再挑戦)。
- **整形を変えた異質メンバーのアンサンブル** — tree(raw)+linear(std) の rank平均が単体超え (EXP002 +0.00221)。
  → 同じ情報でも整形を変えると脱相関し、混合で効く。

## 確認された事実・機序 (新事実)

- **同じ特徴量でもモデルで効き方が逆になる** — `standardize` は木に無効・線形に有効。
  これは「特徴量を族×モデル (models/scale) で分類する」(`feature_taxonomy.py`) の直接の根拠。
  単一の「効く特徴量リスト」では表現できず、**モデル軸を持つ分類が要る**ことの実証。

### 書き方 (テンプレ)

新しい発見に気づいたら、下記の粒度で追記する:

```
- **<一言で言える事実/施策>** — <根拠: EXP番号 / Δ / seed数>。
  → <次にどう活かすか / どの設計判断の根拠か>
  ⚠️失効トリガ: <この前提が崩れたら無効になりうる、という具体条件>
```

- 「効いた施策」= 再利用したい打ち手 (do this)
- 「確認された事実・機序」= 問題そのものについて分かった真実 (why / 天井 / 構造)
- `⚠️失効トリガ` を具体的・反証可能に書く。これが ANTI_PATTERNS の再検証起点になる
- 確信度が上がったら根拠 (EXP番号・Δ・seed数) を必ず添える
